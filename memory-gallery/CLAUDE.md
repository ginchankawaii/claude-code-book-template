## ★ v2 ピボット（2026-07 実運用の結果決定。以降はこちらが正）

v1（連想鎖）を実際に回した本人の結論により、**既定フローをマインドマップ画像生成に変更**した。
本人が手動で検証したフロー（NotebookLMで構造マインドマップ化 → Nano Banana で手描き風に意匠化）の自動化が目的。

```
記憶カードに素材（テキスト or 表スクショ）を追加
  → mindmap.py  Claude が素材から構造化マインドマップを抽出（素材にない事実は入れない）
  → 忠実性チェック（誤りを絵にしない。素通し禁止）
  → render.py   Nano Banana (Gemini) が構造を一字一句保ったまま手描き風イラスト化
  → notion.py   カードのカバー＋本文に自動添付、Mermaid（検証用の正）も本文へ
閲覧 = 記憶カードDBのギャラリービューを眺めるだけ
```

- 実行: `python -m src.main run`（対話）/ `--yes`（無人）/ `--dry-run`（構造抽出まで）
- **v1 連想鎖モードは `--chains` フラグで存続**（アンカー台帳・3案生成・一言の手書き）
- 生成画像は `out/` にもローカル保存される
- 測定ゲート（8章）の「連想鎖だけで測る」前提は本人判断で撤回。以降の章は v1 モードの参考資料として残す

## ★ v3 結線エンジン（ナレッジグラフ化。本人とディスカッションの上合意）

マインドマップの**要所1〜3箇所**に、本人の記憶（アンカー台帳）と既習カードを結線する（`graph.py`）。

- **属性アンカー＝再利用可**（番号・体系ものは同じアンカーで固定キャラ化。例: 図鑑番号=LSAタイプ）
- **感情アンカー＝1項目専有**（台帳の「使用済み項目」relation で管理。使用済みは結線不可）
- **絵に入れてよいのは視覚イメージのみ**（`visual` フィールド。人名・エピソード文字は静的チェックで遮断）。
  連想の説明文（`reason`、人名可）は Notion 本文にだけ記録
- 結線の `reason` 内の技術的断定も LLM 審査（誤った理由づけを絵にしない。素通し禁止）
- 既習カードへの結線は記憶カードDBの「**関連カード**」relation（self-relation・追加済み）に記録し、
  絵にも「関連: カード名」の小さな道標を描く
- 新アンカーの投入規約: 自動提案は必ず**状態=候補**で入れる。**採用への昇格は本人だけ**が行う
  （偽の記憶はアンカーとして機能しないため、機械が勝手に採用してはならない）

### v3.2 「俺の記憶感」強化（本人と合意 2026-07-23）

生成マップに本人の記憶が「使われている感」が出ない問題への対策3点:

- **絵に出してOK 列**（アンカー台帳 checkbox・既定オフ）: **本人がチェックした行だけ**、アンカー名・
  世界観の文字を画像プロンプトに出せる（例: ポケモン・WIN5）。チェックなしは従来通り全遮断。
  機械が新規アンカーにこのチェックを入れることは**絶対にない**（フェイルクローズ維持）
- **テーマ結線**（`scope:"theme"`・最大1個）: 体系アンカーがマップ全体と噛み合うとき、隅の挿絵ではなく
  **絵全体の世界観**（図鑑風ページ等）として描く。2個目以降の theme は spot に格下げ
- **結線を見える場所に書く**: アンカー property=結線アンカー名、連想鎖 property=結線の連想文、
  本文は画像直後に 🔗 callout（Mermaid の下に埋めない）

### v3.1 アンカー発掘インタビュー（`interview.py`・本人と合意 2026-07-23）

台帳に効くアンカー結線が1つも成立しなかったカードでは、**対話モード限定**で本人に質問して
新アンカーをその場で引き出す（`--yes`/`--dry-run` では発動しない）。

- 発動条件: アンカー結線ゼロのときだけ（台帳が育てば質問は自然に減る）
- 質問形式: 誘い水つきの自由回答（一番重要なノード1つ＋ヒント2〜3方向。本人の言葉で答える）
- 本人がその場で語り y 確認した記憶は**状態=採用**で台帳に追加し、即結線して絵に反映する
  （「候補どまり」規約は機械の発明への防波堤であり、本人の口述はこれに当たらない）
- 回答にない事柄の補完は禁止。曖昧な回答はアンカー化しない（fail-closed）
- 挿絵（visual）は新アンカー名込みで静的チェック、reason の技術的断定は LLM 審査を通す
- 感情 select は既存選択肢（報酬/嫌悪/罪悪/痛み/屈辱）以外を台帳に作らない（パース時に検証）
>
> ⚠️ **このリポジトリ／このSPEC以下は共有しない。** 個人名・家族の名前・個人的な出来事を含む。
> ENCOR Study System はメンターにゲスト共有中。**そこにリンクもページも作らないこと。**
> **リポジトリは private 必須**（`anchors.yaml` に個人情報が入るため）。
>
> **v1 の範囲（出力）：画像生成なし。連想鎖とマインドマップ（Mermaid）だけ。**（L3=作画は測定ゲート後）
> **v1 の範囲（入力）：テキストに加え画像も可。** Claude が画像（例：Ping-t問題のスクショ、トポロジ図）を解釈し「覚えたい事実」を抽出する。

Ping-t で繰り返し落ちる項目に、**個人の記憶に既にあるアンカーを経由した連想鎖**を張り、定着させる。

役割分担：

- **Ping-t ＝ 想起エンジン**（既存、コンボ率 95.2% の実績）
- **記憶ギャラリー ＝ 符号化の介入**

ギャラリー単体で完結させない。**必ず Ping-t に戻す。**

## 4. 生成ルール（品質ゲート）

1. **アンカーは必ず台帳から選ぶ。** 台帳外の一般的な連想は使わない。
2. **鎖は覚えたい事実に戻ること。** 「7→イチロー→野球」は逸れて終わりで不可。「7→イチロー→特別枠でメジャーへ→NSSA＝特別扱いのスタブ」なら可。
3. **同じアンカーを別項目に使い回さない。** 衝突すると両方消える。
4. **属性アンカーと感情アンカーを組み合わせる。** 属性だけだと冷たいパズル、感情だけだと戻ってこない。
5. **技術的断定は公式ブループリントと突合する。** 記憶システムでのハルシネーションは致命傷。間違った連想を焼き付けたら、何も作らないほうがマシ。
6. **最後に自分で一言書く。** ダサくていい。ここは絶対に自動化しない（生成効果）。

## 5. アーキテクチャ

```
L1 連想層   Anthropic API   入力(テキスト/画像) → 事実抽出 → アンカー台帳 → 連想鎖を３案（テキスト）
     ↓ ★事実照合ゲート
L2 構造層   コード（非AI）  SVG骨格。ノード位置・矢印・日本語ラベル
     ↓ 骨格を参照画像として渡す
L3 意匠層   画像AI          骨格を保ったまま絵にする
     ↓
Notion       REST API        カバー=画像、本文=Mermaid（検証用の正）
```

**なぜ画像AIにマインドマップを丸投げしないか。**

記憶システムにおけるハルシネーションは致命傷。普通のアプリなら「変な画像が出た」で済むが、ここでは**間違った内容を強烈に定着させる**ことになる。「Type3はASBRが生成」と描かれた美しい絵を10回見たら、それが定着する。**何も作らないほうがマシ。**

だから構造と文字は決定論的に、絵だけ生成AI。**Mermaid版を必ず残す**（検証用の正）。

### 入力モダリティ（テキスト＋画像／v1で対応）

「覚えたいもの」はテキストでも画像でもよい。画像（Ping-t問題のスクショ、トポロジ図など）は L1 で Claude（マルチモーダル）が解釈し、**覚えたい事実を1つ抽出**してから連想鎖に進む。

⚠️ **画像の読み違いはハルシネーション経路。** L3(作画)の誤りと同じく、入口の誤読も「間違った内容を強烈に定着させる」ことになる。抽出した「覚えたい事実」は、連想鎖を張る前に必ず確定させる：

1. 抽出事実をカードに書き出して**人間が確認**、または
2. 公式ブループリントと**突合**（`gate.py`）。

**順序を死守：入力画像 → 事実を正しく抽出できたか確認 → 連想鎖。** 絵を描くのはあくまで測定ゲート後（L3）。

```
https://api.notion.com/v1/
Notion-Version: 2026-03-11
Authorization: Bearer <統合トークン>
```

## 8. ★ 測定ゲート（実装の前にここ）

**EEM カードに「コンボ前 77」を記録済み**（W01で唯一 80% を割った分野、36/47）。

次に EEM を回したときのコンボ率を「コンボ後」に入れる。

- **上がった** → 連想鎖だけで効いている。画像層(L3)は不要かもしれない
- **変わらない** → 画像層を足す根拠になる
- **下がった** → 仕組みごと捨てる

**この１回の測定を挑む前に L3 を実装しないこと。**

## 9. スコープ制限

**本業は ENCOR 合格（2026/09/26）。このシステムは手段であって目的ではない。**

- v1 は **1〜2時間で動くもの**を作って終わり。凝り始めたら本末転倒。
