## HIGH

### H-1. `_pit_recent_mean` の同日 point-in-time リーク
- **場所**: `keiba/features.py:162-174`（`_pit_recent_mean` → `h_recent_relfinish` / `h_form_momentum`、定義は features.py:86-87）
- **問題**: `x.shift().rolling(window)` を `race_date` ソート行に行ベースで適用するため、同一馬が同日に複数走した場合、対象レースより前に確定する「同日の別レースの着順（rel_finish）」を特徴量に取り込む。他の集計（`_pit_daily_cumulative` / `_pit_max`）は日次集計で当日を丸ごと除外しており、この1関数だけが docstring 鉄則2「同日の他レースも混入させない」に違反。`h_form_momentum` も汚染を継承。
- **なぜ重要**: 確定後カラムの同日リーク。合成データで現実に発火（同日複数出走が条件により 102組/4166・209馬日。leak-free 参照実装と最大 138〜211 行が不一致）。リークは「対象レースと同時間帯に決まる強さ信号」を先取りし、バックテストROI・Brier/loglossを実力以上に見せ、本番（地方・複数場・同日連闘）で再現しない楽観バイアスを生む。`docs/RESEARCH_JRAVAN.md` 4.2(B) が最重要視する論点。
- **推奨修正**: 行ベース shift をやめ、`(horse_id, race_date)` で rel_finish を日次集計してから日次系列に `groupby(entity).shift().rolling(window)` を当て、`(entity, race_date)` で元行へ merge（`_pit_daily_cumulative` と同じ当日除外パターン）。`h_form_momentum` は修正後 recent を使えば自動修正。
- **確信度**: high
- **実行検証**: 済（3観点が独立に再現確認）

### H-2. `monte_carlo_ruin` が単勝の相互排他を無視（破産確率が楽観バイアス）
- **場所**: `keiba/betting.py:137-179`
- **問題**: race_id を一切参照せず、各ベット勝敗を独立ベルヌーイ `rng.random() < probs[idx]` で再サンプル。単勝は同一レース内で相互排他（高々1頭しか勝てない）だが、独立サンプリングでは同一レースの複数ベットが同時的中しうる（ベットの約55%が複数ベット最大5点のレースに属し、5点×p=0.3 で同時2点以上的中の確率≈0.47）。
- **なぜ重要**: 現実に不可能な余剰的中でバンクロール経路を系統的に有利化し、`ruin_prob` 下方・`median_final`/最大DD を楽観化。research 6.3 章で破産確率/DDのMC事前把握が資金管理の核心とされる目的を損なう。
- **推奨修正**: レース単位で勝者を高々1頭選ぶカテゴリカル抽選にし、賭けた馬が勝者かを判定。または実 `is_win` のブートストラップへ切替。独立ベルヌーイを残すなら非現実性をメトリクス名と docstring に明記。
- **確信度**: high
- **実行検証**: 済

### H-3. headline ROI（フラット136%・ケリー148%）の選択バイアス・少フォールド分散が未開示
- **場所**: `keiba/pipeline.py:format_report`（79-81行）／`keiba/__main__.py:27`／`keiba/README.md:18-20`
- **問題**: 既定デモが「フラットROI 136.6%／ケリーROI 148.9%／最終資金 2.842x」を大きく表示するが、真の form エッジが無い `w_form=0` でも EV フィルタ + isotonic 較正のノイズ選択だけで flatROI 27〜115%（seed 9/11/21）・33〜115ベットが発生。既定設定でも seed 2-11 で flatROI 88.9〜137.8%（平均111.8%, sd15.0%）とシード依存大。caveat は「控除率の壁」にしか触れず、選択バイアス + 少フォールド分散による上振れに言及がない。
- **なぜ重要**: 科学的誠実さで最も危険な誤読点。研究docが警告する「ノイズへの過剰適合・選択バイアス」をデモ自身が体現しているのに読者に示されず、趣味開発者が合成デモROIを skill 由来の超過収益・下限値と誤読し、実データで控除率20%の壁を越えられると誤期待しうる。
- **推奨修正**: `format_report` に「この回収率は合成デモ値で (a)EVフィルタの選択バイアスと (b)少フォールド分散を含む。エッジ無しでもノイズ選択で100%超が出る」旨を明記。参考値として `w_form=0` の flatROI、複数シードの ROI 分布（平均±sd）を併記し単点でなく分布で提示。READMEの「myopia 0.3 →市場に勝てない」も myopia 0 でベットが出てROIがゼロにならない点を補足。
- **確信度**: high
- **実行検証**: 済

## MEDIUM

### M-1. `audit_temporal_invariance` が同日リークを構造的に検出できない
- **場所**: `keiba/leakage.py:26-70`（`truncated = runners[runners.race_date <= rdate]`、leakage.py:58）
- **問題**: 「対象レース日以前（同日含む）」に切り詰め未来"日"削除のみを検証するため、同日兄弟レース依存（H-1）を原理的に検出できず `audit['ok']=True` を返す。`test_temporal_invariance_no_leakage` / `test_pit_aggregates_use_only_past` が誤って緑になる。実データで H-1 が実在するのに mismatches=[] を確認済。
- **なぜ重要**: リーク検出の最後の砦が甘く、H-1 がテストを素通り。「監査済みだから安全」という誤った安心を与え、実データ投入後の新たな同日リークも検知できない。
- **推奨修正**: 監査に「同日兄弟レース削除に対する不変性」チェックを追加（同日 `race_date==rdate` かつ `race_id != rid` を削除した truncated でも対象行特徴が一致するか検証）。`race_date < rdate` 版での再現確認も補助追加。対象列に `h_recent_relfinish` / `h_form_momentum` を含める。修正後 recent が sibling有=1.0 → 削除=NaN で mismatch 顕在化を確認済。
- **確信度**: high
- **実行検証**: 済

### M-2. walk_forward で `step_days < test_days` 時に test 窓重複→二重計上
- **場所**: `keiba/backtest.py:57,66`
- **問題**: `step = wf.step_days or wf.test_days` で `step_days` を `test_days` 未満にできるが、その場合 test 窓が隣接フォールドと重複。`test_pred_frames` / `all_bets` を単純連結するため、重複区間のレース・ベットが pooled に二重計上され、settle_flat/settle_kelly/monte_carlo_ruin/quality が全て重複データで計算される。ガード・重複排除なし。
- **なぜ重要**: 既定（step=test_days）は安全だが、フォールド数を増やすため step を小さくする（walk-forward では一般的）と、ROI・n_bets・破産確率・log-loss が静かに二重計上で汚染。
- **推奨修正**: pooled の `preds_df` / `bets_df` を `(race_id, horse_id)` で重複排除、または `step_days < test_days` を assert で禁止、または各 test 窓の非重複を検証。
- **確信度**: high
- **実行検証**: 済

### M-3. `--myopia 0` の「市場に勝てない正直な結果」表現が楽観的
- **場所**: `keiba/__main__.py:27-28`（--myopia ヘルプ）／`keiba/README.md:18-19`／`keiba/synth.py:50-52`
- **問題**: `--myopia 0` を「効率的（エッジ無）」「市場に勝てない正直な結果」と表現するが、実測では myopia=0.0 でも 3〜154 ベットを選び、ROI 0〜88%、平均ブレンド重み 0.78〜0.90（≈1.0 ではない）。「エッジ無=ベットが出ない/ROIが払戻率に収束」という素直な挙動にならない。
- **なぜ重要**: log-loss 基準では正しいが、ベッティング層はノイズ選択でベットを出すため、「効率市場でもベットが出る=エッジがある」または「myopia 0 でも勝てた」と取り違える余地。
- **推奨修正**: ヘルプ/READMEに「myopia=0 では予測エッジ（log-loss改善）は出ないが、EVフィルタは較正ノイズで少数ベットを出すことがある。回収率はおおむね払戻率(~80%)±選択分散に収束」と補足。
- **確信度**: high
- **実行検証**: 済

- **場所**: `keiba/betting.py:101-134`
- **問題**: 同一レース内の複数ベットが `sort_values(['race_date','race_id'])` 後の逐次ループで決済され、`stake = stake_frac * bankroll` が各ベット後に更新される。同時決定点であるべきレース内ベットの2番目以降が1番目の結果で更新されたバンクロールを基準にサイズされ、相互排他の単勝を「独立機会」として順次複利運用。ベットの約55%が複数ベットのレースに属する。
- **なぜ重要**: ステークがベット順序（=元の行順）に依存し、headline の Kelly ROI/DD に効く（中規模影響）。
- **推奨修正**: 決定点（同一 race_date+race_id）単位でステークを賭け時バンクロールから一括算出し、レース内払戻を合算後にバンクロール更新。1決定点を1ステップとして複利。
- **確信度**: medium
- **実行検証**: 済

## 誤検出の可能性

今回のレビュー結果に**明確な誤検出は含まれない**が、解釈に注意が必要な項目を以下に整理する。

- **H-3 / M-3（headline ROI とmyopia=0 の表現）**: これらは「コードのバグ」ではなく**ドキュメント・科学的誠実さの問題**である。ROI計算ロジック自体は確率・統計観点のレビューで正しいと実行検証されている（kelly/EV/分数ケリー・Harville・較正は順列ブルートフォースと <1e-12 で一致）。したがって「ROIの数値が間違っている」という意味での欠陥ではなく、「合成デモの数値を skill 由来エッジと誤読させる開示不足」が本質。修正は計算変更でなく注記・分布提示で足りる。重大度 high は誤読リスクの大きさによるもので、コード欠陥としての緊急度ではない点に留意。

- **L-4（破産確率の楽観性）**: 唯一 `verified_by_running: false`。docstring には「model_prob を真と仮定」と既に明記されており、設計上の前提を出力側に再掲するかという**表示方針の問題**。H-2（相互排他無視）とは独立だが同じ `monte_carlo_ruin` に起因するため、H-2 修正時にまとめて対応すれば良く、単独では誤検出すれすれの低優先。

- **N-1 後段の `horse_age_days += 0`**: レビュー自身が「コメントで簡略化と明示・実害なし」と認めており、現時点ではバグではない（将来の潜在リスクの指摘にとどまる）。

なお、複数観点が**重複して指摘**した項目は統合済み（H-1 はデータリーク／コード品質の2観点、M-1 は同2観点、N-2/wmean はバックテスト／確率・統計の2観点が独立に同一指摘）。重複は誤検出ではなくむしろ確度を高める材料として扱った。

## 最優先で直すべき上位3件

1. **H-1：`_pit_recent_mean` の同日 point-in-time リーク**（features.py:162-174）
   システムの最重要設計目標「リーク防止」を唯一破る本物の確定後情報リーク。バックテスト全指標を楽観方向に汚染し本番で再現しない。日次集計+当日除外への統一で修正。

2. **M-1：`audit_temporal_invariance` の同日リーク検出盲点**（leakage.py:26-70）
   H-1 とセットで必須。これを直さない限りテストが緑のままリークが温存され「監査済み=安全」という誤った安心が残る。H-1 の回帰防止網として同時修正すべき。

3. **H-2：`monte_carlo_ruin` の単勝相互排他無視**（betting.py:137-179）
   資金管理の核心である破産確率/DDを系統的に楽観化。レース単位カテゴリカル抽選（または実 is_win ブートストラップ）への変更で修正。L-1・L-4 も同じベッティング層なので併せて着手すると効率的。

（補足：H-3 は high だが計算欠陥ではなく開示の問題のため、コード修正の緊急度では上位3件の次点。リリース前のドキュメント整備として必ず対応すること。）
