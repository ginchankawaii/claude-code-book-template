# AI活用による業務効率化戦略 — 東北支店(ドラフト)

追加原価の削減・ドキュメント品質の平準化・残業削減を、AIを活用した
「ツール化」で実現するための戦略メモ。管理職への説明にも使える形でまとめる。

## 1. 基本方針: プロンプト共有ではなくツール共有

| | プロンプト共有 | ツール共有(本方針) |
|---|---|---|
| 共有されるもの | 知識(使い方は人次第) | 能力(誰が使っても同じ動き) |
| 再現性 | 同じ入力でも結果が揺れる | 決定的(同じ入力→同じ結果) |
| 利用者のAIライセンス | 必要 | **不要**(作るときだけAIを使う) |
| 品質保証 | 使う人のスキルに依存 | テストで担保できる |

- AIは「ツールを量産する3Dプリンター」として使う。利用者には普通の
  Pythonスクリプト/PowerShellとして配る
- 既存のプロンプト収集運動は否定しない。集まったプロンプトは
  「誰が何の業務で困っているかのカタログ」= **ツール化候補リスト**として使う

## 2. プロダクト1: 構成チェッカー(追加原価の防止)

→ 実装: [tools/config-checker/](../tools/config-checker/)

### 課題

受注後の追加原価が多発。実例:
- WLCパフォーマンスライセンス不足(既存WLC収容前提の案件で、既存側の
  ライセンス余力を誰も確認しなかった=「知らなかった」型の失敗)
- ASAのライセンス不足でHAが組めなかった

「知らなかった」型の失敗には、注意喚起(チェックリスト)は無力。
**知識そのものがツールに入っている**必要がある。

### 解決策

BOM(CCW見積エクスポート)をルールDBと照合するリンター。

- **判定型ルール**: BOM内で白黒つくもの(例: ASA 2台にライセンス1本 → 警告)
- **質問型ルール**: BOMのパターンから「そこに無いもの」を検出し、既存環境の
  確認を強制する(例: APがあるのにWLCが無い → 既存WLCの機種・AP数・
  ライセンス余力を回答必須の質問として出力)
- SEはレビュー提出**前**にセルフチェックとして実行し、質問への回答を
  前提条件としてレビュー提出物に添付する

### 副次効果: ドキュメント品質の標準化

質問型ルールへの回答が、そのまま「前提条件・見積条件」の標準フォーマットに
なる。現在は人によって粒度がバラバラな前提条件が、ルールDBの質問に答える
形式に揃う。BSC/QMS資料への転記もこの回答から行える。

### 知識の集め方(属人化の解消)

「ベテランの頭の中を言語化する」を一大プロジェクトにはしない。失敗する。

1. **追加原価が発生したら必ずルールを1件追加**してから清算する
   (ソフトウェア開発の「バグには必ず回帰テストを書く」文化の移植)
2. ベテランには白紙で聞かず、**ドラフトルールへの赤入れ**を頼む
   (人は白紙には書けないが、間違った具体物には訂正を入れたくなる)
3. レビューで指摘が出たら、その場でルール化する(指摘1件=ルール1件)

### 価値の測り方

追加原価の実額(円)で語る。過去事例のBOMを食わせて「このツールがあれば
受注前に検出できた」を実演し、当時の追加原価額を横に添える。

## 3. プロダクト2: 工数登録アシスタント(毎日15分の削減)

→ 実装: [tools/outlook-ins-tagger/](../tools/outlook-ins-tagger/)

### 課題

全SEが毎日10〜15分を工数登録(HCM+クラウドログ+Excel連携)に使っている。
クラウドログには「OutlookにINS番号を入れておけば一発連携」の公式機能が
あるが、事前入力の手間がネックで使われていない。

### 解決策

Outlookの予定にINS番号を自動付与するPowerShellスクリプト
(キーワード辞書ベース・ドライラン既定・工数集計のチートシート付き)。

- 社内システムを直接操作しない(公式連携の**入口だけ**を自動化)ため、
  壊れにくく規程面でも安全
- Windows標準機能のみで動作。インストール不要で誰にでも配布できる

### 価値の測り方

毎日10分の削減 × SE 10人以上 × 年220営業日 ≒ **年間370時間以上**。
残業削減目標に直接効く数字として語れる。

## 4. 資産はツールではなくルールDB

1年後にはさらに優れたAIが出ており、ツール(コード)は書き直せばよい消耗品。
時間とともに価値が増える資産は **rules.yaml(支店の失敗知識の構造化DB)** と
**INS辞書のような業務データ**の方。AIが賢くなるほど、構造化された知識の
活用の幅は広がる。
