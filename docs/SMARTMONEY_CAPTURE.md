# 賢い金(中間オッズ)キャプチャと検証設計

## 背景 — なぜこれだけが残ったエッジ候補か

5.5年の実データ検証で「**静的な最終オッズ**では市場に勝てない」が確定した
(単勝・馬連・ワイド・三連複すべて、控除後ROIの95%CI下限が100%を割る)。

唯一エビデンスのある例外が「**締切直前のオッズの動き(賢い金/スマートマネー)**」。

> 東京大学マーケットデザインセンター, arXiv:2509.14645 (2025)
> JRA-VANの5分刻み中間オッズ(2004–2023の全中央競馬)で
> 「最終オッズは情報を完全には集約しておらず、締切直前にオッズが急落した馬
> (=金が入った馬)は、最終オッズが同じ他馬より高い回収率を示す」。
> さらに中間の動きを入れると favorite-longshot bias が弱まる。

ただしこれは**統計的関係**であって、控除率(約20–30%)を引いた回収率100%超の
検証済みシステムではない。**まだ誰も決着をつけていない問い**であり、本プロジェクトで
自前データを貯めて検証する。

重要な事実: 過去の中間オッズはDBに無い(realtimeでしか取れない)。だから
**これから貯める**しかない。realtime の単複オッズは本体DBの **`TS_O1`** に入る
(列に `TanOdds` / `Umaban` / `HassoTime`(発走時刻) / `CollectedAt`(取得時刻)を持つ。
紛らわしい `TS_SOKUHO_O1` は別名の空テーブルで未使用)。

## キャプチャ設計 — 容量を絶対に溢れさせない

`jltsql` の realtime オッズは `0B30`(全賭式)しか無く、三連単まで全部の中間
オッズが入る(1開催で数百万行=容量爆弾)。そこで:

1. `0B30` 込みで取得(単複の時系列が欲しいので不可避)
2. **セッション後に単複(`TS_O1`)だけ `odds_history.db` へ「1分間引き」で退避**
   (realtimeは数秒ごとに再取り込みし `TS_O1` が猛烈に増えるため、(レース×馬番×分)
   単位に間引く。賢い金は秒単位不要、論文も5分刻み)
3. 本体の `TS_O1..O6`(と空の `TS_SOKUHO_O1..O6`)を一掃(`tools/prune_realtime_odds.py`)
4. fetchキャッシュも掃除

`VACUUM` はしない: 21GBのVACUUMは同等の空き容量を要し危険。DELETEは縮めないが
空きページは翌週再利用されるため、本体は**高々1開催分だけ増えて頭打ち**。
`odds_history.db` は単勝だけなので **年100–200MB程度**にしかならない。

土日(開催日)だけ自動で動く。平日は一切動かさない。

## 検証設計(データが数ヶ月貯まってから / エビデンス反映)

`odds_history.db` の `TS_O1` スナップショット列(`TanOdds`/`Umaban`/`HassoTime`/
`CollectedAt`)から特徴量を作り、既存の
`keiba` パイプライン(walk-forward + 95%CIブートストラップ)に載せる。

### 特徴量(各馬・各レース)
各スナップショット時刻 t で単勝暗黙確率 `p_t = (1/odds_t) / Σ(1/odds)`(レース内正規化)。
締切時刻 T(発走時刻)からの相対で:

- **drift_late** = `p(T-1min) − p(T-10min)` … 締切直前の確率上昇(=金が入った)。論文の核心。
- **drift_slope** = 直近K点の `p_t` の傾き(回帰係数)
- **rank_jump** = 人気順位が直近で上がった数
- **level** = 最終オッズ(favorite-longshot との交互作用のため必ず併用)

論文の主張(締切直前に急落=高回収)を最も忠実に表すのは `drift_late`。

### 賭けルール(Chapman / Ziemba を反映)
- **極端な大穴を除外**: 推定勝率 `p < 0.04` は捨てる(Chapman 香港: これで回収改善)。
- **オーバーレイのみ薄く**: EV閾値超のみ、分数ケリーで小さく(Ziemba Dr Z)。

### 検証プロトコル(まやかし対策)
- walk-forward(過去→未来)。**最終オッズを必ずコントロール**に入れ、
  「軌跡が最終オッズに**上乗せ**する予測力」を測る(最終オッズに既に入った分の二重計上を防ぐ)。
- 控除後ROIの**95%CIブートストラップ下限>100%**で初めて「黒」と判定。
- ルックアヘッド注意: 軌跡は締切後に完全観測。「シグナルが在るか」を測る段階は
  締切1分前まででよい(論文と同条件)。**実運用(締切前に賭ける)は別問題**で、
  その場合は「締切n分前までの部分軌跡」で同じ検証をやり直す必要がある。

### 必要サンプルの目安
中央競馬は1開催日 約36レース。土日で約72レース、月 約8開催日 ≈ 290レース。
賭けは候補の一部なので、**判定に足る数百ベットを貯めるには概ね1シーズン(3–6ヶ月)**。
それまでは「貯める」フェーズ。途中経過は随時 95%CI の幅が縮むのを見て判断する。

## まとめ
- これは**唯一まだ決着のついていない**エッジ候補。成功保証は無い。
- 仕組みは軽量・自動・容量安全。あとは時間が貯めてくれる。
- 数ヶ月後、`odds_history.db` を入力に検証を回し、95%CI下限が100%を超えるかを見る。
